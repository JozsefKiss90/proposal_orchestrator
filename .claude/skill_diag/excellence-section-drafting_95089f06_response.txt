      "content": "MAESTRO's methodology is structured as a co-designed research programme in which three tightly integrated pillars — neuro-symbolic planning (WP2), adaptive memory architecture (WP3), and decentralised multi-agent coordination (WP4) — share common data representations, evaluation benchmarks, and integration interfaces from the outset, unified by the cross-cutting External Tool and API Orchestration Layer. This co-design philosophy is the source of the project's distinctive architectural contribution: the pillars are not sequential work packages but simultaneous, mutually constraining research streams whose interfaces are specified jointly in the first 18 months and integrated from Month 18 onwards. The methodology proceeds across three phases: architecture design and foundational research (Months 1–18), implementation, integration, and pilot deployment (Months 18–36), and full demonstrator validation and exploitation (Months 36–48), with structured 6-month integration checkpoints throughout.\n\nPillar 1 — Neuro-Symbolic Planning and Reasoning (WP2, lead: ATU)\n\nThe core methodological concept is the hybridisation of LLM-based flexibility with the formal guarantees of symbolic planning. The approach is motivated by the demonstrated fragility of pure LLM chain-of-thought planning over long task horizons: plans degrade unpredictably because LLMs optimise for local plausibility without global constraint satisfaction and cannot recover from intermediate failures without full restart. MAESTRO's solution is a hybrid architecture in which symbolic methods provide structural correctness and LLM models provide task-specific heuristics.\n\nTask T2-01 (Sub-task graph decomposition with formal pre/post-conditions) designs the formal specification language for executable sub-task graphs — specifying the pre-conditions, post-conditions, and verification checkpoints constituting a verifiable plan — and develops the LLM-based decomposition model trained to produce structurally valid plan representations. Task T2-02 (MCTS and HTN planning integration with LLM heuristics) integrates Monte Carlo Tree Search (MCTS) and hierarchical task networks (HTN) with LLM-driven heuristic evaluation: symbolic algorithms provide the search structure and guarantee formal correctness while the LLM provides task-specific heuristic estimates guiding search efficiency across large goal-decomposition spaces. Task T2-03 (Plan monitoring and self-correction) implements runtime monitoring that detects precondition violations and execution failures, triggers replanning from the last verified intermediate state rather than restarting the full plan, and propagates failure information to the adaptive memory architecture (WP3) for experience-based future plan selection — addressing plan fragility at the architectural level rather than through prompt engineering.\n\nThe principal assumption is that hybrid neuro-symbolic plan structures can capture the semantics of a sufficiently broad class of real-world tasks to demonstrate generalisation across the healthcare, manufacturing, and logistics demonstrator domains. A significant challenge is designing formal specification languages expressive enough for real-world task diversity without inducing computational intractability in plan search; this is addressed by the hierarchical decomposition strategy in T2-01, which constrains the search space at each decomposition level, and by the LLM heuristic guidance in T2-02, which focuses search on tractable portions of the goal space. National or international activities whose results feed into WP2 include prior work on MCTS-based game-playing agents and HTN planning in autonomous robotics, which provide mature algorithmic foundations that Pillar 1 extends and hybridises with LLM reasoning.\n\nPillar 2 — Adaptive Memory Architecture (WP3, lead: BIIS)\n\nThe core methodological concept is that effective long-horizon agent behaviour requires memory that structurally differentiates three qualitatively distinct memory functions managed by a unified controller, analogous to established cognitive models of human memory systems. The tri-store architecture designed in Task T3-01 provides: working memory (a structured, attention-managed short-term store maintaining the agent's current goals, active plan state, and recent observations — enabling coherent plan execution within a single task horizon); episodic memory (a retrieval-augmented store of past task episodes indexed by context similarity, enabling experience-based agent behaviour without retraining — drawing on retrieval-augmented generation advances in which BIIS holds significant expertise); and semantic memory (a persistent knowledge graph connecting domain concepts, tool capabilities, and task patterns, continuously updated through agent interactions and queryable by the planning and coordination pillars).\n\nTask T3-02 (Memory indexing and retrieval) develops sparse and dense retrieval strategies for episodic memory and graph-based query mechanisms for semantic memory, optimised for the retrieval latency constraints of real-time agent interaction loops. Task T3-03 (Forgetting mechanism and coherence maintenance) develops learned forgetting policies balancing recall accuracy against computational cost and coherence maintenance mechanisms detecting and resolving contradictions between memory stores — addressing the practical challenge that growing episodic and semantic stores can introduce inconsistency over extended interactions. The assumption that a unified memory controller can learn effective information-routing policies across the three stores through training on accumulated interaction traces is validated progressively through the demonstrators, where long-horizon clinical diagnostic sequences (WP5) and multi-actor logistics planning (WP7) provide challenging long-horizon evaluation scenarios.\n\nPillar 3 — Decentralised Multi-Agent Coordination (WP4, lead: CERIA)\n\nThe core methodological concept is that formally specified coordination protocols, rather than ad-hoc LLM-based agent communication, are necessary to achieve reliable, accountable, and convergent multi-agent behaviour in deployment settings where failures carry real consequences. CERIA's expertise in formal multi-agent systems, game theory, and mechanism design — evidenced by an ERC Advanced Grant on formal methods for multi-agent systems and participation in 50+ EU collaborative projects — provides the methodological foundation.\n\nTask T4-01 (Task-delegation and negotiation protocol design) defines a formal task-delegation protocol based on shared commitment structures, where agents decompose tasks, negotiate subtask assignments, and establish binding delegation contracts with provable convergence properties under partial observability conditions representative of the demonstrator deployment environments. Task T4-02 (Convergence and conflict resolution under partial observability) develops conflict resolution mechanisms for situations where agent plans interfere, shared resources are contested, or partial observations produce inconsistent world models, with formal convergence proofs derived under bounded communication and observation constraints. Task T4-03 (Auditable multi-agent interaction trace module) implements a structured trace system recording every delegation, negotiation step, and joint decision in a human-inspectable format, stored in the shared adaptive memory architecture (WP3) and available to human operators in all demonstrator settings — a critical requirement for trustworthy deployment in safety-relevant environments including clinical decision support and industrial process control. The challenge of achieving formal convergence guarantees in heterogeneous agent settings where capability distributions are not uniform is addressed by the mechanism design approach in T4-01, which is specifically formulated to accommodate heterogeneous agent types.\n\nCross-Cutting Capability — External Tool and API Orchestration Layer\n\nThe External Tool and API Orchestration Layer is architecturally integrated across all three pillars and addresses the call's essential requirement (SR-04, scope_requirements.json) for principled external tool and API integration to overcome limitations of large AI models. The planning engine (Pillar 1) determines when and which tools to invoke based on formal tool selection policies embedded in the typed tool registry; the memory architecture (Pillar 2) stores tool capabilities, execution histories, and learned tool-selection heuristics in the semantic memory store, enabling the agent to improve tool selection over successive interactions; and the coordination layer (Pillar 3) governs shared tool access across collaborating agents, manages delegated tool invocation, and resolves conflicts when multiple agents require the same external resource simultaneously.\n\nThe layer provides: a typed tool registry and capability model with machine-readable input/output schemas, pre-conditions, post-conditions, cost profiles, and reliability metadata, queryable and updatable at runtime; formal tool selection and invocation policies integrated with the neuro-symbolic planning engine; standardised API adapters abstracting heterogeneous protocols — REST, GraphQL, HL7 FHIR for healthcare, OPC-UA for manufacturing — into a uniform invocation interface enabling cross-domain tool reuse; post-invocation verification validating tool outputs against expected schemas and domain constraints before incorporating results into agent state, grounding agent reasoning in verified external data; execution monitoring with timeout detection, error classification, and automated fallback strategies (alternative tool selection, graceful degradation, human-in-the-loop escalation); and auditable tool execution traces providing full provenance for every invocation — recording the requesting agent, planning rationale, invocation parameters, raw response, validation outcome, and downstream effect on agent state. This architecture makes the MAESTRO approach categorically distinct from agent systems that treat tool use as a simple function-calling interface.\n\nDemonstrators and Validation (WP5, WP6, WP7, WP8)\n\nAll three research pillars feed into three demonstrators co-designed with sector-leading industry partners and validated in operationally realistic settings targeting TRL 5. The healthcare demonstrator (WP5, lead: NIHS) deploys the integrated MAESTRO framework for clinical decision support, using HL7 FHIR connectors to access NIHS's de-identified clinical data repository (3M+ patient records, ethics-approved) and validating against 500 benchmark cases across 5 diagnostic domains with clinical expert assessment. The manufacturing demonstrator (WP6, lead: ISIA) deploys multi-agent process optimisation in ISIA's sensor-instrumented Industry 4.0 pilot factory with digital twin simulation environment and OPC-UA industrial connectors, validating across a 3-month operational period against the KPI targets specified in OBJ-5. The logistics transfer demonstrator (WP7, lead: ELI) validates cross-sector generalisation using ELI's operational logistics data spanning 30+ countries, testing agent-based planning and negotiation across three international supply chain corridors with real disruption scenarios. WP8 (lead: FIIT) produces the cross-cutting evaluation framework (D8-01, Month 12), integrates all pillars into the unified MAESTRO system (T8-02), and releases the community benchmark suite (D8-02, Month 42) and final integration report (D8-03, Month 48), enabling community reproduction and extension of results.\n\nInterdisciplinary Approach\n\nMAESTRO's methodology is intrinsically interdisciplinary, integrating expertise from seven qualitatively distinct research and practice communities into a unified research programme. Formal methods and theoretical computer science (convergence proofs, formal plan specification) are provided by CERIA, whose ERC Advanced Grant background ensures methodological rigour in the formal protocol design. Machine learning, neuro-symbolic AI, and LLM architectures are provided by ATU (H2020 trustworthy AI project, AI-on-demand platform contribution, national AI competence centre) and BIIS (H2020 knowledge infrastructure, HE unified transformers project, national hybrid intelligence programme). Clinical informatics and health data interoperability are provided by NIHS (HE European Health Data Space pilot, H2020 AI for clinical imaging diagnostics, national precision medicine initiative), ensuring that the healthcare demonstrator design reflects realistic clinical workflows and regulatory constraints. Smart manufacturing, digital twin modelling, and Industry 4.0 integration are provided by ISIA (HE AI for manufacturing process optimisation, H2020 big data for smart factory, EIT Manufacturing partnership), ensuring that the manufacturing demonstrator is validated against genuine industrial requirements. Logistics operations and AI-powered supply chain optimisation are provided by ELI (H2020 logistics platforms for European supply chains, Innovation Lab with 50+ dedicated R&D staff), whose operational data spans 80+ countries. LLM development, AI productisation, and open-source release management are provided by BAL (HE European large language models, EuroHPC pre-exascale supercomputer access), ensuring that the framework is engineered for production deployment and community release. AI evaluation, responsible AI assessment, technology readiness assessment, and EDIH-mediated technology transfer are provided by FIIT (HE AI testing and evaluation frameworks, H2020 AI-on-demand platform contribution, regional EDIH operations and EDIH network connections across 6 EU member states).\n\nThe integration of social sciences and humanities perspectives is embedded in the demonstrator design: the clinical decision support demonstrator (WP5/NIHS) incorporates human-AI interaction and clinical workflow considerations in its expert validation design (T5-03), and the auditable trace requirements across WP4–WP7 reflect responsible AI principles regarding transparency and accountability in human-AI collaborative systems, particularly in healthcare and industrial settings where autonomous agent decisions carry operational consequences.\n\nGender Dimension in Research and Innovation Content\n\nMAESTRO's core research develops AI agent architectures and coordination protocols whose fundamental algorithmic properties are not themselves gender-specific. However, the gender dimension is material to the research content in two demonstrator contexts. In the healthcare demonstrator (WP5), the benchmark of 500 de-identified clinical cases across 5 diagnostic domains (OBJ-4) will be designed with demographic representativeness, including sex/gender stratification across case selection, to ensure that diagnostic reasoning accuracy targets are met equitably across patient subgroups — addressing the known risk that AI diagnostic systems can exhibit differential accuracy across demographic groups, particularly in clinical domains where presentation and prevalence differ by sex or gender. The adequacy of this stratification will be evaluated in the clinical expert validation phase (T5-03/NIHS). In the integrated evaluation framework (WP8), FIIT's expertise in responsible AI assessment and certification will be applied to evaluate whether demonstrator outputs exhibit differential performance across protected characteristics including sex and gender, with findings feeding back into the training data curation and evaluation design processes in WP3 and WP8. These gender-dimension considerations are integrated into the demonstrator validation methodology and the evaluation framework specification (D8-01, Month 12), not treated as supplementary additions.\n\nOpen Science Practices\n\nMAESTRO commits to comprehensive open science practices throughout the project. The core MAESTRO agent framework, trained models, benchmark datasets, the External Tool and API Orchestration Layer specification, and all demonstrator evaluation results will be released as open-source software through WP9 (lead: BAL): Task T9-01 manages open-source release management and community building; T9-02 manages exploitation strategy and IP management (including licensing decisions for the core framework and any patentable innovations arising from the formally specified coordination protocols or the typed tool registry design). The open-source strategy is explicitly designed to lower barriers for European SMEs and startups and to reduce strategic dependency on non-European AI platforms, consistent with the European Apply AI Strategy.\n\nAll communicable research outputs will be contributed to the AI-on-demand platform (SR-10, scope_requirements.json), and results will be validated through Testing and Experiment Facilities with dissemination through European Digital Innovation Hubs via FIIT's EDIH network (connections across 6 EU member states). A Data Management Plan (DMP, D1-02, WP1/ATU, due Month 6) will cover data categorisation, access conditions, retention policies, anonymisation requirements, and FAIR compliance for all project datasets including: the ethics-approved de-identified clinical case benchmark (NIHS, 500 annotated cases); the industrial process data collected from ISIA's pilot factory (subject to industrial confidentiality provisions); ELI's anonymised logistics operational data (spanning 30+ countries); and all model artefacts, training datasets, and evaluation benchmarks produced by WP2–WP4. The DMP will be maintained as a living document updated at each 6-month integration checkpoint and will serve as the basis for the more detailed exploitation and dissemination plan (D9-01, WP9/BAL, due Month 6).
