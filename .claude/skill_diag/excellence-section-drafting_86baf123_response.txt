      "content": "MAESTRO's methodology is grounded in its central design principle: that reliable, deployable AI agents require the co-design of planning, memory, coordination, and tool orchestration — not their post-hoc assembly. The methodology is structured across five functional layers corresponding to the project's nine-WP architecture.\n\nLAYER 1 — FOUNDATIONAL RESEARCH PILLARS (WP2, WP3, WP4)\n\nThe three foundational work packages run in parallel from month 1. WP2 (ATU-led) implements OBJ-1 — Neuro-symbolic planning engine for autonomous task decomposition — through three tasks: T2-01 designs the sub-task graph decomposition mechanism with formally specified pre-conditions and post-conditions; T2-02 integrates MCTS and HTN planning algorithms with LLM-based heuristic evaluation to explore solution spaces efficiently; T2-03 implements plan monitoring and self-correction, detecting execution failures and replanning from verified intermediate states. Deliverable D2-01 (Neuro-symbolic planning architecture specification, month 18) formally specifies the design prior to software implementation (D2-02, month 36), enabling rigorous verification before demonstrator integration. WP3 (BIIS-led) implements OBJ-2 — Unified adaptive memory architecture — through tri-store memory design (T3-01), memory indexing and retrieval strategy implementation (T3-02), and forgetting mechanism and coherence maintenance (T3-03). The memory architecture design specification D3-01 (month 12) anchors the formal architecture; the adaptive memory system software D3-02 is delivered at month 30. WP4 (CERIA-led) implements OBJ-3 — Decentralised multi-agent coordination protocols — through formal protocol design (T4-01: task-delegation and negotiation protocol), convergence and conflict resolution under partial observability (T4-02), and auditable multi-agent interaction trace module implementation (T4-03). Deliverable D4-01 (Multi-agent coordination protocol specification, month 18) delivers the Formally verified multi-agent coordination protocol suite (OUT-3), constituting the primary formal-methods contribution of WP4; deliverable D4-02 (Decentralised coordination software, month 36) provides the reference implementation.\n\nLAYER 2 — CROSS-CUTTING TOOL ORCHESTRATION (WP2, WP3, WP4, WP8)\n\nThe External Tool and API Orchestration Layer (OBJ-8) is co-designed with all three research pillars as an architectural element, not a separate capability. The planning engine (WP2) integrates formal tool selection and invocation policies as a first-class component of plan generation, so that tool invocation decisions are verifiable and traceable. The memory architecture (WP3) stores tool capability metadata, execution histories, and learned tool-selection heuristics in the semantic memory store — enabling experience-based tool selection without retraining. The coordination layer (WP4) governs shared tool access across collaborating agents and resolves conflicts when multiple agents require the same external resource. WP8 integration tasks (T8-02, FIIT-led) combine all three pillars and the orchestration layer into the unified MAESTRO system, producing the MAESTRO benchmark suite dataset (D8-02, month 42) and final integration and evaluation report (D8-03, month 48). The evaluation framework and benchmark specification D8-01 (month 12) is produced early to enable aligned KPI tracking across all WPs from project start.\n\nLAYER 3 — SECTOR DEMONSTRATORS (WP5, WP6, WP7)\n\nThree demonstrator work packages validate MAESTRO capabilities in operational environments. Each follows a three-task pipeline: use case definition and benchmark design, system deployment and integration, and expert validation. WP5 (NIHS-led, OBJ-4 — Healthcare demonstrator — clinical decision support) deploys an AI agent for clinical diagnostic reasoning with HL7 FHIR integration, drawing on NIHS's de-identified clinical data repository (3M+ records, ethics-approved). WP6 (ISIA-led, OBJ-5 — Manufacturing demonstrator — process optimisation) deploys a multi-agent manufacturing process optimisation system with OPC-UA industrial connectors and digital twin APIs in ISIA's Industry 4.0 pilot factory. WP7 (ELI-led, OBJ-6 — Logistics transfer demonstrator — cross-sector generalisation) deploys agent-based logistics planning across three international supply chain corridors using ELI's operational logistics data and supply chain simulation environment. All three demonstrators receive the planning engine software prototype (D2-02, month 36), adaptive memory system software (D3-02, month 30), and decentralised coordination software (D4-02, month 36) as inputs, in accordance with the WP dependency map.\n\nLAYER 4 — INTEGRATION AND BENCHMARKING (WP8, FIIT-led); LAYER 5 — DISSEMINATION (WP9, BAL-led)\n\nWP8 (OBJ-7 — Open framework release and ecosystem contribution, jointly with WP9) unifies research pillar outputs into the integrated MAESTRO system and releases the community benchmark suite. WP9 manages the dissemination and exploitation plan (D9-01, month 6), the open-source software release (D9-02, month 36), and the final dissemination and impact report (D9-03, month 48).\n\nKEY CHALLENGES AND METHODOLOGICAL MITIGATIONS\n\nLLM plan fragility over long task horizons is addressed by replacing LLM heuristics as the primary control signal with formal HTN and MCTS planning with verification checkpoints and monitored self-correction (WP2). Memory coherence under resource constraints is addressed by the learned memory controller dynamically balancing between working, episodic, and semantic stores based on task context (WP3). Multi-agent convergence under partial observability is addressed by CERIA's formal protocol design with mathematical convergence proofs, grounded in CERIA's ERC Advanced Grant expertise in formal multi-agent methods (WP4). Heterogeneous API integration across three industrial domains is addressed by the standardised External Tool and API Orchestration Layer with domain-specific adapters for HL7 FHIR (healthcare), OPC-UA (manufacturing), and logistics tracking and customs clearance APIs.\n\nNATIONAL AND INTERNATIONAL R&I ACTIVITIES\n\nMAESTRO builds on prior EU-funded work: ATU's Horizon Europe contribution to the AI-on-demand platform; BIIS's Horizon Europe project on unified transformers for text and retrieval; CERIA's ERC Advanced Grant on formal methods for multi-agent systems; NIHS's Horizon Europe European Health Data Space pilot implementation project; ISIA's Horizon Europe project on AI for manufacturing process optimisation; FIIT's Horizon Europe project on AI testing and evaluation frameworks. MAESTRO links to AI Factories and Data Labs for compute-intensive training and evaluation (scope requirement SR-11), the AI-on-demand platform for communicable results dissemination (SR-10), and Testing and Experiment Facilities for demonstrator validation.\n\nINTERDISCIPLINARY APPROACH\n\nMAESTRO integrates four disciplinary traditions that do not typically interact: (1) LLM-based AI agent engineering and production deployment (ATU — neuro-symbolic AI, automated planning; BAL — LLM development, MLOps); (2) formal methods, automated planning theory, and multi-agent systems (ATU, CERIA — game-theoretic coordination, formal verification); (3) neural information retrieval, knowledge representation, and memory architectures (BIIS — retrieval-augmented generation, knowledge graph embedding); and (4) domain-applied engineering informatics — clinical informatics and health data integration (NIHS), industrial engineering and digital twins (ISIA), logistics AI and supply chain operations (ELI). The interdisciplinary integration is structural, not additive: the planning engine (WP2) must interface with the memory architecture (WP3) through defined memory controller APIs; the coordination layer (WP4) draws on both formal protocol theory and distributed AI; and the tool orchestration layer requires API design patterns from all three industrial domains simultaneously. Integration is managed through WP8 with shared deliverable ownership and 6-month integration checkpoints.\n\nSOCIAL SCIENCES AND HUMANITIES (SSH)\n\nMAESTRO is primarily a technical AI systems project. No dedicated SSH disciplinary research pillars are planned, and no claim to SSH disciplinary contributions is made.
