Current LLM-based agents degrade unpredictably over extended task sequences, cannot maintain coherent state across context-window boundaries, lack principled mechanisms for multi-agent coordination, and treat external tool and API integration as an afterthought rather than a core architectural requirement. These limitations prevent the deployment of truly autonomous AI systems in high-value sectors of the European economy.\n\nMAESTRO resolves these limitations through three mutually reinforcing research pillars — neuro-symbolic planning and reasoning, adaptive memory architecture, and decentralised multi-agent coordination — unified by a cross-cutting External Tool and API Orchestration Layer, co-engineered from the outset rather than assembled from independent research components.       "content": "MAESTRO employs a structured three-phase iterative research methodology implemented across nine work packages. Phase 1 (months 1–18) establishes foundational co-designed architectures: formal specifications for the planning engine (WP2, ATU), memory architecture (WP3, BIIS), coordination protocols (WP4, CERIA), and External Tool and API Orchestration Layer (ATU) are co-designed with the cross-cutting evaluation framework and integration interface contract (D8-01, FIIT, month 12) specified before individual pillar implementations reach prototype stage — ensuring integration conformance is continuously verifiable throughout development rather than deferred to project end. Phase 2 (months 12–36) realises the co-designed architecture in implemented software: WP2 produces the neuro-symbolic planning engine prototype (D2-02, month 36), WP3 the adaptive memory system (D3-02, month 30), and WP4 the decentralised coordination software (D4-02, month 36). Phase 3 (months 18–48) co-designs and validates sector-specific demonstrators: WP5 (NIHS), WP6 (ISIA), and WP7 (ELI) begin sector-specific co-design from month 18 using early pillar outputs, completing operational validations by month 42 and culminating in the final integration and evaluation report (D8-03, FIIT, month 48).\n\nThe methodology is grounded in four complementary conceptual models. For the neuro-symbolic planning engine (Pillar 1, WP2, ATU — tasks T2-01: sub-task graph decomposition with formal pre/post-conditions; T2-02: MCTS and HTN integration with LLM heuristics; T2-03: plan monitoring and self-correction), the fundamental model holds that neural flexibility and symbolic formal guarantees are complementary rather than competing. LLMs provide heuristic evaluation at decision nodes within Monte Carlo Tree Search and hierarchical task network algorithms, contributing commonsense reasoning while the symbolic structure enforces verifiable pre-conditions and post-conditions at each sub-task step. Plan monitoring detects execution failures and triggers replanning from the most recently verified intermediate state — preserving computational work and preventing failure propagation across long task horizons.\n\nFor the adaptive memory architecture (Pillar 2, WP3, BIIS — tasks T3-01: tri-store architecture design; T3-02: memory indexing and retrieval strategy; T3-03: forgetting mechanism and coherence maintenance), the foundational model holds that effective long-horizon agent behaviour requires three qualitatively distinct and interdependent memory subsystems: working memory (structured short-term store for active goals and current plan state), episodic memory (retrieval-augmented store indexed by context similarity enabling experience-based behaviour without retraining), and semantic memory (persistent knowledge graph connecting domain concepts, tool capabilities, and task patterns, continuously updated through agent interactions). A learned memory controller governs information flow between stores — deciding what to commit to long-term memory, what to retrieve, and when to forget — balancing recall accuracy against computational cost. This architecture goes substantially beyond retrieval-augmented generation by integrating all three store types under a unified learned controller.\n\nFor the decentralised multi-agent coordination protocols (Pillar 3, WP4, CERIA — tasks T4-01: task-delegation and negotiation protocol design; T4-02: convergence and conflict resolution under partial observability; T4-03: auditable multi-agent interaction trace module), the foundational model holds that effective agent collaboration requires formally specified mechanisms with provable properties: a task-delegation protocol with provable convergence enabling agents to decompose tasks, negotiate subtask assignments, and establish shared commitment structures; conflict resolution mechanisms for plan interference and resource contention under partial observability; and structured auditable traces as a first-class output of every delegation and negotiation step. These formal properties distinguish MAESTRO from LLM-based ad-hoc agent communication that lacks convergence guarantees and accountability.\n\nFor the External Tool and API Orchestration Layer (cross-cutting, ATU), the model holds that tool use must be governed by the planning engine, tracked by the memory architecture, and coordinated across agents by the coordination layer. The typed tool registry expresses tool capabilities, preconditions, postconditions, cost profiles, and reliability metadata in machine-readable form that the planning engine reasons over when constructing invocation policies. Post-invocation verification validates tool outputs against expected schemas and domain constraints before they are incorporated into agent state. Execution monitoring provides timeout detection, error classification, and automated fallback strategies — alternative tool selection, graceful degradation, human-in-the-loop escalation — integrated with the planning engine's self-correction capability. Auditable tool execution traces are stored in the adaptive memory architecture for experience-based tool selection and in the multi-agent coordination layer for delegation accountability.\n\nThe central methodological challenge is integration: each pillar requires data representations and invocation APIs from the others, and this cannot be resolved post hoc. MAESTRO addresses this through a design-before-integrate strategy enforced by WP8's integration interface contract (D8-01, month 12, FIIT). This contract specifies shared data representations, evaluation benchmarks, and integration APIs before individual WPs complete prototype implementations. WP8 task T8-02 (MAESTRO unified framework integration, FIIT) verifies conformance continuously from month 12, ensuring that the final integration activity (months 42–48) validates running integrated systems rather than requiring crisis convergence engineering.\n\nA second challenge is operational demonstrator co-design. Research prototypes must be adapted to real sector environments that impose constraints — healthcare data privacy regulations, manufacturing real-time latency requirements, logistics multi-actor trust boundaries — not visible in benchmark evaluation alone. MAESTRO addresses this by co-designing demonstrators with sector-leading industry and clinical partners from month 18. NIHS provides a de-identified clinical data repository of 3M+ patient records (ethics-approved) and a clinical simulation environment validated against real diagnostic pathways. ISIA provides an Industry 4.0 pilot factory with sensor-instrumented production lines and a digital twin simulation environment.
